Sync System Redesign Plan — Delta Sync + Unified SyncContext
Context
The current sync system calls fetchAndUpdateLocal() which hits getAll* on the server
and wipes + replaces the entire local IndexedDB table every sync cycle. With growing data
this is slow, wastes bandwidth, and risks data loss. The 5 separate sync hooks each run their
own interval, lock, and listener — causing race conditions and redundant re-renders.

This plan redesigns the sync system with two major architectural improvements:

Delta sync — only fetch records created/updated since the last sync
Unified SyncContext — one central orchestrator replaces 5 fragmented hooks
All 8 bugs from the previous analysis are folded in and fixed as part of this redesign.

Without updatedAt on Category, Product, and SalesReturn, delta sync cannot know which
records changed. Without soft delete, deleted records disappear without a trace.

Frontend gap analysis

No lastSyncedAt tracking per entity — every fetch is a full fetch
5 hooks each register their own window.addEventListener('online', ...) — 5x listeners
5 hooks each run a setInterval — 5 independent timers firing at different phases
Stale closure in interval → isSyncing guard never fires → overlapping syncs
Event listener leak in cleanup() — removeEventListener uses wrong reference

Step C: Add delta query logic to each service


// category-management.service.ts
async getAllCategories(updatedAfter?: string) {
  const where: Prisma.CategoryWhereInput = {
    deletedAt: null,  // always exclude soft-deleted
  };

  if (updatedAfter) {
    where.updatedAt = { gte: new Date(updatedAfter) };
  }

  const categories = await this.prisma.category.findMany({ where });

  // Also return IDs of records deleted since updatedAfter
  let deletedIds: string[] = [];
  if (updatedAfter) {
    const deleted = await this.prisma.category.findMany({
      where: { deletedAt: { gte: new Date(updatedAfter) } },
      select: { id: true },
    });
    deletedIds = deleted.map(d => d.id);
  }

  return { data: categories, deletedIds };
}
Step D: Update delete methods to soft-delete


// BEFORE (hard delete):
async deleteCategory(id: string) {
  await this.prisma.category.delete({ where: { id } });
}

// AFTER (soft delete):
async deleteCategory(id: string) {
  await this.prisma.category.update({
    where: { id },
    data: { deletedAt: new Date() },
  });
}
Same pattern applies to Product, StockIn, StockOut, SalesReturn services.

incremental responses are small.

4.1 New file: frontend/src/context/SyncContext.jsx
State shape:


{
  isSyncing: false,
  syncPaused: false,          // true when 401 interceptor fires
  lastSyncedAt: {             // per entity, from sync_metadata table
    categories: null,
    products: null,
    stockIns: null,
    stockOuts: null,
    salesReturns: null,

    products: 0,

    salesReturns: 0,
  },
  errors: {                   // per entity error message

    products: null,

    stockOuts: null,
    salesReturns: null,
  },
  lastSyncCompleted: null,    // Date of last full successful sync
}
Exposed via useSyncContext():


{
  // state fields above, plus:
  triggerSync,        // () => Promise — run full sync cycle (respects lock)
  forceSync,          // () => Promise — bypass lock, run immediately
  pauseSync,          // () => void   — called by auth contexts on 401
  resumeSync,         // () => void   — called by auth contexts after re-auth
  refreshCounts,      // () => Promise — update pendingCounts without syncing
}
Internal implementation outline:


export const SyncProvider = ({ children }) => {
  const { isOnline } = useNetworkStatusContext();
  const isSyncingRef = useRef(false);      // ref for stale-closure-safe interval
  const syncLockRef = useRef(null);        // master promise lock
  const intervalRef = useRef(null);

  const [state, setState] = useState({ /* shape above */ });

  // ----- Orchestration order (respects dependency chain) -----
  const runFullSyncCycle = async () => {
    try {
      await categorySyncService.syncCategories();
      await productSyncService.syncProducts();
      await stockInSyncService.syncStockIns();
      await stockOutSyncService.syncStockOuts();
      await salesReturnSyncService.syncSalesReturns();
      await refreshCounts();
      await loadLastSyncedTimestamps();
    } catch (err) {
      // per-entity errors already stored by services
    }
  };

  // ----- Sync lock (single lock for all entities) -----
  const triggerSync = useCallback(async () => {
    if (!isOnline || isSyncingRef.current || state.syncPaused) return;
    if (syncLockRef.current) { await syncLockRef.current; return; }

    let unlock;
    syncLockRef.current = new Promise(res => { unlock = res; });
    isSyncingRef.current = true;
    setState(s => ({ ...s, isSyncing: true }));

    try {
      await runFullSyncCycle();
      setState(s => ({ ...s, lastSyncCompleted: new Date() }));
    } finally {
      isSyncingRef.current = false;
      syncLockRef.current = null;
      unlock();
      setState(s => ({ ...s, isSyncing: false }));
    }
  }, [isOnline, state.syncPaused]);

  // ----- Online event + interval (only ONE of each, not 5) -----
  useEffect(() => {
    if (isOnline) triggerSync();

    intervalRef.current = setInterval(() => {
      if (isOnline && !isSyncingRef.current) triggerSync();
    }, 30_000);

    return () => clearInterval(intervalRef.current);
  }, [isOnline]);  // re-registers when online status changes

  // ----- 401 pause/resume -----
  const pauseSync  = useCallback(() => setState(s => ({ ...s, syncPaused: true })), []);
  const resumeSync = useCallback(() => {
    setState(s => ({ ...s, syncPaused: false }));
    triggerSync();  // immediately retry after re-auth
  }, [triggerSync]);

  return (
    <SyncContext.Provider value={{ ...state, triggerSync, forceSync, pauseSync, resumeSync, refreshCounts }}>
      {children}
    </SyncContext.Provider>
  );
};

Part 5 — Modified fetchAndUpdateLocal() (Delta Sync Core)
The key change in all 5 sync services. Pattern shown for categorySyncService.js:


async fetchAndUpdateLocal() {
  // Read last sync timestamp from IndexedDB metadata table
  const meta = await db.sync_metadata.get('categories');
  const lastSyncedAt = meta?.lastSyncedAt || null;

  // --- Fetch from server (outside any transaction) ---
  let result;
  try {
    result = await categoryService.getAllCategories(lastSyncedAt);
    // result = { data: [...], deletedIds: [...] }
  } catch (fetchError) {
    console.error('[categories] Fetch failed — local data preserved:', fetchError);
    return;  // bail: never touch IndexedDB on network failure
  }

  const { data: updatedRecords, deletedIds = [] } = result;

  // Guard: if first-time full fetch returned nothing, don't wipe
  if (!lastSyncedAt && (!Array.isArray(updatedRecords) || updatedRecords.length === 0)) {
    console.warn('[categories] Empty full-fetch response — skipping to preserve local data');
    return;
  }

  // --- Apply changes atomically ---
  await db.transaction('rw', db.categories_all, db.synced_category_ids, db.sync_metadata, async () => {
    if (!lastSyncedAt) {
      // First-ever sync: replace all
      await db.categories_all.clear();
      await db.categories_all.bulkPut(
        updatedRecords.map(c => ({ id: c.id, name: c.name, description: c.description, updatedAt: c.updatedAt, lastModified: new Date() }))
      );
    } else {
      // Delta sync: upsert changed records
      for (const record of updatedRecords) {
        await db.categories_all.put({ id: record.id, name: record.name, description: record.description, updatedAt: record.updatedAt, lastModified: new Date() });
      }
      // Remove server-deleted records from local DB
      for (const id of deletedIds) {
        await db.categories_all.delete(id);
        const mapping = await db.synced_category_ids.where('serverId').equals(id).first();
        if (mapping) await db.synced_category_ids.delete(mapping.localId);
      }
    }

    // Update last-synced timestamp
    await db.sync_metadata.put({
      entity: 'categories',
      lastSyncedAt: new Date().toISOString(),
      lastFullSyncAt: !lastSyncedAt ? new Date().toISOString() : (meta?.lastFullSyncAt || null),
    });
  });
}
This single pattern change:

Fixes Bug 1 (data wipe) — no .clear() on delta path
Fixes Bug 3 (shouldFetchFresh) — naturally replaced by timestamp check
Fixes the core performance problem — fetches only changed data
The same pattern applies identically to all 5 services, adjusted for their fields.

API service layer changes (e.g., categoryService.js):

Part 9 — Migration Strategy (handles users with stale data)
Existing users already have data in IndexedDB from full fetches.
The sync_metadata table won't exist yet — db.sync_metadata.get('categories') returns undefined.
When lastSyncedAt is null/undefined, the code falls into the "first-time full fetch" path.

So for existing users, the first sync after deploy behaves exactly like today (full replace)
but with the empty-array guard. After that, the metadata timestamp is written and all future
syncs are incremental. Zero user impact.

Dexie version bump (15 → 16):
Dexie runs its upgrade hook automatically. No data is lost when adding a new table.
The sync_metadata table starts empty, which triggers the full-fetch fallback.
